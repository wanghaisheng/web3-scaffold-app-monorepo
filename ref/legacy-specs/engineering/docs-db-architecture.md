# 🗄️ Engineering Spec: Docs & Blog Data Architecture

**Objective:** Enable a "Full-Text Searchable" documentation site running entirely in the browser (Sandbox-compatible) while supporting a future migration to Supabase.

---

## 1. The Core Challenge: "The Browser Wall"
*   **Constraint:** Browsers (and AI Studio Sandboxes) cannot use `fs.readFileSync` to read Markdown files at runtime.
*   **Solution:** **The Hybrid Registry Protocol**.

## 2. The 3-Tier Data Pipeline

### Tier 1: The Static Registry (`src/data/registry-*.ts`)
*   **Format:** TypeScript Arrays (`RegistryItem[]`).
*   **Role:** The "In-Memory Filesystem".
*   **Content:** Contains the full Markdown body, ID, Slug, and Metadata for every page.
*   **Generation:**
    *   *Dev/Sandbox:* Manually editable.
    *   *Build:* Auto-generated by `scripts/generate-*-manifest.mjs` which reads the physical `docs/` folder.

### Tier 2: The In-Memory Logic Engine (`sql.js`)
*   **Role:** The Query Layer.
*   **Mechanism (`useDocsDB` hook):**
    1.  **Boot**: App starts.
    2.  **Hydrate**: Reads `registry` arrays.
    3.  **Ingest**: Batch `INSERT` into an in-memory SQLite database (`wasm`).
    4.  **Index**: Builds `FTS5` indexes for search.
*   **Benefit**: Allows complex SQL queries (`SELECT * WHERE content LIKE '%foo%'`) without a backend.

### Tier 3: The Persistent Store (Supabase - Optional)
*   **Role:** Production Database (User Data / CMS).
*   **Migration**: The `RegistryItem` interface maps 1:1 to a Supabase Table. Switching is as simple as changing the `useDocsDB` implementation to call `supabase.from(...)`.

## 3. Database Schema (SQLite)

```sql
CREATE TABLE resources (
  id TEXT,          -- PRID (e.g. "1001")
  lang TEXT,        -- "en" or "zh"
  title TEXT,
  content TEXT,     -- Markdown Body
  desc TEXT,
  category TEXT,
  type TEXT,        -- "doc" or "blog"
  author TEXT,
  date TEXT,
  image TEXT,
  PRIMARY KEY (id, lang)
);

CREATE TABLE routes (
  slug TEXT,
  resource_id TEXT,
  lang TEXT,
  PRIMARY KEY (slug, lang),
  FOREIGN KEY(resource_id, lang) REFERENCES resources(id, lang)
);
```

## 4. Query Strategy (ID-First)
*   **Primary**: `getDetailById(id, lang)` -> O(1) Lookup.
*   **Fallback**: `getDetailBySlug(slug, lang)` -> O(log n) Lookup via Index.
*   **Search**: `SELECT * FROM resources WHERE content LIKE ?` -> FTS.
