# ✍️ 增长规约：文案优化策略

**核心信条：** “视觉靠画，逻辑靠说，全球发布。”

## 1. 策略：从“指挥”到“启航”
我们将所有文案与一个具体、可操作的工作流对齐。用户购买的不是“管理”；他们购买的是“结果”（一个已发布的产品）。

### 关键转向
*   **旧钩子：** “停止编码。开始指挥。” (抽象)
*   **新钩子：** “将你的 AI 草图变成全球 SaaS。” (具体)
*   **原因：** 用户想知道他们*得到*什么。“全球 SaaS”是一个有价值的资产。“指挥”只是工作。

## 2. G+L+C 叙事结构
每一份营销文案都必须强化这个三步仪式。

### G: Google AI Studio (视觉)
*   **关键词：** “愿景”、“草图”、“自由大脑”、“像素级完美”。
*   **痛点：** “我能想象它，但我写不出代码。”
*   **解决方案：** 上传截图 -> 获取代码。

### L: Lovable (逻辑)
*   **关键词：** “脊梁”、“后端”、“数据库”、“支付”、“自然语言”。
*   **痛点：** “我有一个漂亮的前端，但它什么也做不了。”
*   **解决方案：** “将此表单连接到 Supabase” -> 完成。

### C: Cloudflare (发布)
*   **关键词：** “躯体”、“全球边缘”、“零配置”、“即时”。
*   **痛点：** “部署很难。服务器很贵。”
*   **解决方案：** Git 推送 -> 全球上线。

## 3. 语气规则
1.  **扮演“老兵”**：使用经验丰富的老兵找到捷径的形象。这能建立信任。
2.  **攻击“维护陷阱”**：不断提醒用户本地环境的痛苦（npm install、版本冲突）。
3.  **颂扬“作弊”**：将效率框定为一种不公平的优势或套利。“既然能说，何必编码？”

## 4. CTA 优化
*   **漏斗顶部：** “获取协议” (好奇心)
*   **漏斗底部：** “开始启航” (行动)
*   **避免：** “了解更多” (被动)

## 5. “客户语言”协议 (功能翻译)
**规则：** 使用“客户语言”沟通，而非“功能列表”。
**目标：** 内容必须围绕“这解决了什么问题？”或“客户成功故事”来组织，而不是罗列技术规格。

| ❌ 技术功能 (内部) | ✅ 客户语言 (外部) |
| :--- | :--- |
| “React 19 + TanStack Router + Vite” | “无需配置服务器，即可发布高性能应用。” |
| “自动生成 JSON-LD Schema” | “无需聘请 SEO 机构，即可在 Google 上自动排名。” |
| “Supabase 行级安全” | “您的用户数据安全且达到企业级标准。” |

*   **社会认同优先：** 展示其他构建者的推荐（例如，“教练 X 如何节省了 20 小时”）比列出 50 个功能更有效。
*   **结构：** 每个功能块必须与一个具体的**“减轻的痛苦”**或**“实现的成果”**配对。
