# Use-case 写作指南（场景页 / 人群价值页）

Use-case 的目标是“把某类人/某类场景怎么用、用完得到什么”讲清楚。没有可核验的真实客户数据时，用 use-case；拿到真实验证后再升级成 testimonial。

## 适用场景

- 面向特定角色（放射科/采购/经销/工程师等）或具体任务
- 强调收益、流程与边界，不强行塞“成功故事”

## 推荐结构

```markdown
# [Persona] 用 [Product/Capability] 做到 [Outcome]

首段：一句话说明场景、痛点与结果（含边界）

## Who this is for
- 适用人群/机构
- 不适用情况

## The scenario
用 1–2 段写清起点、约束与成功标准

## Benefits
5–7 条：每条尽量量化（或给出衡量方式）+ 引用/口径

## How it works
3–5 步：输入/处理/输出/失败处理

## Common pitfalls
3–5 条：真实限制与规避方式

## FAQ
5–7 个问题
```

## 写作要点

### 不要编造“客户成功”
- 没有可核验数据就不写“某医院提升 X%”；改成“我们建议用这些指标衡量”。
- 如果必须举例，明确标注为示例/假设，避免误导。

### 用“可衡量指标”替代形容词
- 把“效率更高”写成“用扫描量/报告周转时间/返工率等衡量”。

### EEAT（Experience + Expertise）
- 描述流程时写清边界条件：依赖什么数据、需要哪些前置条件、失败怎么办。

## Humanizer 检查（Use-case 常见雷区）

- 不要套“挑战-解决方案-未来展望”模板。
- 列表不要每条同句式；混合长短句，允许更口语的提醒句。

## 验收清单

- 首段明确“谁/什么场景/结果/边界”。
- Benefits 至少 3 条有量化或衡量方式，并有口径/来源。
- How it works 可复现，且包含失败处理或限制说明。
- 无虚构归因、无口号式总结。
