# iOS App 首次上架 Checklist

> 基于实际踩坑经验整理，涵盖元数据、订阅、功能测试、审核策略和发布流程。逐项打勾，避免"5连拒"。

---

## 一、提交前：元数据与账户信息

### 截图要求
- [ ] **不包含价格、折扣等文字**（第一次被拒原因）
- [ ] 截图清晰展示 App 核心功能
- [ ] 不同尺寸设备截图完整

### 必填信息
- [ ] **支持网址（Support URL）**：填写有效、可访问的 URL
- [ ] **App 描述**：如果涉及订阅/条款，需在描述中写明《用户协议》关键内容（第五次被拒原因）
- [ ] **隐私政策（Privacy Policy）**：
  - [ ] 必须在 App 内可点击查看
  - [ ] 链接有效且可访问
- [ ] **使用条款（Terms of Use）**：
  - [ ] App 内必须有明显的按钮/链接，点开能完整显示
  - [ ] App Store 页面描述里也要包含条款的核心内容（第五次被拒重点）

### 设备兼容性
- [ ] **App 类别与设备**：即使设为 iPhone only，也要测试 iPad 兼容模式（审核方会在 iPad 上运行）

---

## 二、订阅与 IAP（应用内购买）

### 测试账号
- [ ] **提供"过期订阅的测试账号"**：Apple 需要验证续订、过期、恢复逻辑（第三次被拒原因）

### 订阅流程
- [ ] **订阅流程完整无报错**：从发起购买、支付成功、到服务器回调，全程不能崩溃或报错（第四次被拒核心）
- [ ] **恢复购买按钮**：必须存在且可用
- [ ] **订阅价格与 App Store 产品一致**：不要在代码里写死价格
- [ ] **Terms & Privacy 按钮在 IAP 流程前后均可点击**：Apple 会用 iPad 测试按钮响应区域

### IAP 测试场景
- [ ] 新用户首次订阅
- [ ] 订阅续订
- [ ] 订阅过期
- [ ] 恢复购买
- [ ] 取消订阅
- [ ] 网络异常情况下的处理

---

## 三、功能与稳定性（"能不能用"）

### 真机测试
- [ ] **所有按钮在真实设备上可点**：不要只依赖 TestFlight 或模拟器（第二次/第三次被拒原因）
- [ ] **登录/注册流程完整**：无白屏、无死循环
- [ ] **后端服务稳定**：审核期间不要改部署、不要停机（第四次被拒因付费接口报错）
- [ ] **测试设备多样性**：至少用 iPhone + iPad（不同尺寸、系统版本）各一台真机测试

### 异常处理
- [ ] **处理网络异常**：不直接崩溃，显示友好提示
- [ ] **处理超时情况**：设置合理超时时间，显示加载状态
- [ ] **处理空数据**：后端返回空数据时的 UI 展示
- [ ] **处理权限拒绝**：相机、相册等权限被拒时的处理

### 核心功能
- [ ] 主要功能流程完整可用
- [ ] 数据持久化正常
- [ ] 推送通知正常（如有）
- [ ] 深度链接正常（如有）

---

## 四、审核策略与心态

### 理解审核本质
- [ ] **审核员会以最少步骤、最快速度"用坏"你的 App**，不是只检查"有没有"
- [ ] 审核员会在不同设备、不同系统版本上测试
- [ ] 审核员会尝试各种边界情况和异常操作

### 拒绝升级规律
- [ ] **第1轮** → 元数据问题（截图、描述、链接等）
- [ ] **第2-3轮** → 功能/法律问题（条款、隐私、基本功能）
- [ ] **第4-5轮** → 真实使用流程崩溃、隐藏条件缺失

### 拒绝后处理
- [ ] **不要只改被指出的点**：主动用 iPad 重走一遍完整用户路径
- [ ] 完整测试路径：注册 → 付费 → 过期 → 恢复
- [ ] 仔细阅读拒绝原因，理解背后的问题
- [ ] 检查相关功能，不仅限于被指出的具体问题

### 时间管理
- [ ] **预留 2-4 天审核周期**：节假日（如清明）会延长
- [ ] 避免在节假日前后提交
- [ ] 提前准备好所有材料和测试

### 情绪管理
- [ ] **被拒时允许崩溃 10 秒**，然后按清单逐项排查
- [ ] 保持理性，不要情绪化回复审核团队
- [ ] 把每次拒绝当作改进机会

---

## 五、最后一步：真正的"发布"

### 发布流程
- [ ] **审核状态变为 Ready for Distribution 后，必须手动点击"Set Up Availability"**（发布到商店）
  - [ ] 否则 App 会显示 "Removed from Sale"
- [ ] **发布后请用另一部设备（非开发者账号）实际下载、订阅一次**
- [ ] 验证发布版本与测试版本功能一致

### 发布后验证
- [ ] 在 App Store 搜索并下载
- [ ] 完整走一遍用户流程
- [ ] 验证订阅功能正常
- [ ] 检查页面描述、截图是否正确显示

---

## 六、AI 编程特别提醒

### 代码质量
- [ ] **不要完全依赖 AI 写代码后直接上传**
- [ ] AI 建议你"手动改某一行"时，往往是因为你该理解自己的代码逻辑
- [ ] 审核被拒后的修复过程，才是真正学会"让系统可靠"的机会

### 测试覆盖
- [ ] AI 生成的代码需要充分测试
- [ ] 边界情况和异常处理不能依赖 AI 自动完成
- [ ] 理解每一行关键代码的作用

## 八、常见被拒原因速查

| 被拒原因 | 检查项 | 优先级 |
|---------|-------|--------|
| 截图包含价格 | 检查所有截图 | 高 |
| Support URL 无效 | 测试链接可访问性 | 高 |
| Privacy Policy 缺失 | App 内可点击查看 | 高 |
| Terms of Use 不完整 | App 内 + 描述都要有 | 高 |
| iPad 不兼容 | iPad 模式测试 | 高 |
| IAP 流程报错 | 完整走一遍购买流程 | 高 |
| 缺少测试账号 | 提供过期订阅账号 | 中 |
| 按钮不可点 | 真机测试所有按钮 | 中 |
| 网络异常崩溃 | 异常处理测试 | 中 |
| 后端服务不稳定 | 审核期间保持稳定 | 高 |
